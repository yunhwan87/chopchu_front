# 2026년 3월 9일 오전 - 요청 관리 탭 기능 구현 및 패치 작업 내역

## 1. 개요
기존 `CommunicationScreen`에 **"보낸 요청, 받은 요청, 해결 완료"** 탭 형식을 차용하여 협업 툴 형태의 팀원 간 "요청 주고받기" 및 "답변 스레드" 기능을 새롭게 개발하고 관련된 이슈들을 픽스했습니다.

## 2. 주요 개발 내역

### 2.1 Database Schema (Supabase)
- **`requests` 테이블 생성**: 프로젝트 내 특정 유저(요청자)가 특정 유저(담당자)에게 보내는 요청글의 기본 정보 관리.
- **`request_messages` 테이블 생성**: 해당 요청 내에서 댓글/스레드 방식으로 여러 번 답변을 남길 수 있도록 구조화.
- **상태값 (Status)**: `'pending'`, `'in_progress'`, `'need_info'`, `'resolved'` 네 가지 상태값을 ENUM이나 CHECK 구문으로 설정.

### 2.2 프론트엔드 API (`src/api/requests.js`)
- `fetchRequests`: **내가 보낸 요청(sent)**, **내가 받은 요청(received)**, **해결된 요청(resolved)** 으로 분기하여 데이터를 쿼리.
- `fetchRequestMessages`: 특정 요청글 내부의 달린 스레드(답변)를 시간순으로 가져오기.
- `addRequestMessage`: 입력 폼에서 전달받은 값으로 새 댓글 작성하기.
- `updateRequestStatus`: 요청글의 상태를 변경하도록 구현.
- **조인 관계 수정 (Foreign Key 명시)**: 여러 개의 외래키가 동일한 `profiles` 테이블을 참조하여 조인 에러가 발생하던 문제를 `profiles!requests_sender_id_fkey` 형태로 제약조건 이름을 직접 명시하여 해결.

### 2.3 프론트엔드 UI (`CommunicationScreen` 등)
- **필터 탭**: UI에서 '받은 요청', '보낸 요청', '해결 완료' 버블 메뉴를 통해 상태값을 바꿔가며 화면을 표시하도록 작업.
- **`CommunicationLog` 모듈식 컴포넌트**: 전체 요청들의 간략한 제목, 보낸 사람, 상태 배지를 보여주는 리스트 구현.
- **`RequestDetailModal` (상세 뷰)**: 카드를 클릭 시 뜨는 모달창. 기존에 없던 요청의 세부 텍스트, 댓글(스레드) 입출력을 볼 수 있도록 추가.
- 하단 네비게이션에 플러스(+) 작성 버튼이 가려지던 UI 겹침 이슈 수정.

### 2.4 세부 로직 및 사용성(UX) 패치
- **보내는 분/받는 분 렌더링**: 
  - 만약 회원님이 팀원들을 이메일 초대로 추가하여 닉네임 설정이 되어 있지 않더라도, `null` 에러를 뱉는 대신 유저의 **이메일 주소를 우선 표기**하도록 예외처리(Fallback) 구현.
- **자동 진행 로직**:
  - '요청 확인 대기 중(pending)'인 요청의 담당자가 상세 팝업 창을 열어 내용을 확인하면, 자동으로 '해결 중(in_progress)'으로 상태가 변경되도록 구현.
- **요청 해결 및 복구 권한 (요청자 시점)**:
  - 본인이 보낸 요청(요청자)에 대해 직접 "해결 완료 표시"를 누르면 최종 종료가 되고, 잘못 눌렀거나 추가 지원이 필요할 경우 다시 "해결 되돌리기" 버튼을 눌러 상태를 롤백할 수 있는 권한 작업 완성.

## 3. 핵심 트러블슈팅 포인트
- **RLS (Row Level Security) 설정 오류 디버깅**:
  - Supabase의 기존 `profiles` 테이블에 **"자신의 프로필만 볼 수 있음"** 이라는 보안 정책이 걸려있어 요청 리스트 조회 시, 타인의 이름이나 정보가 비어(null)서 넘어오던 현상을 파악.
  - 해결: 접속한 모든 유저가 최소한 다른 팀원의 닉네임/이메일 등 기초 프로필 정보를 열람할 수 있도록 `profiles`의 SELECT 정책을 완화하는 쿼리를 작성하여 반영.
